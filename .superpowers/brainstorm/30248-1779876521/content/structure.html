How should the site be structured?

The site currently fits on one scrollable page. With download links, self-hosting, features, and a roadmap — we have choices about where things live.

A

One long page — extended

Keep everything in index.html. Add a Roadmap section, expand the Self-host section, add direct download buttons. Visitors scroll through: Hero → Features → Download → Self-host → Roadmap → FAQ. Fast to build, easy to maintain.

Pros

  • Zero routing complexity
  • Fastest to ship
  • Works with the current static site setup

Cons

  • Self-hosting docs can get long on the same page
  • Can't deep-link to e.g. /install
B

Landing page + separate /install page

Index stays focused (hero, features, roadmap, quick download). A dedicated install.html (or /install/) holds the full self-hosting guide — Azure AD steps, env vars, deployment options. Nav links to it.

Pros

  • Self-host docs can be thorough without overwhelming newcomers
  • Clean deep links to /install
  • Mirrors SELF_HOSTING.md naturally

Cons

  • One more file to maintain
  • Slightly more nav to wire up
C

Multi-page site (landing + install + changelog)

Three pages: landing, a full install/self-host guide, and a changelog/releases page. Nav across all three.

Pros

  • Most organized for growth
  • Changelog can auto-pull from GitHub releases eventually

Cons

  • Most work
  • Changelog becomes stale unless automated
